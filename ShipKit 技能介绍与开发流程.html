项目总览

这是按 .docs/new-shipkit-design/ 从零实现的新 ShipKit 设计。它不修改、不依赖旧 skills/ 目录；所有新技能和配套脚本都放在 new-skills/ 下。

实现根目录：new-skills/
版本：new-shipkit-v1
核心原则：简化而不简陋、单一事实源 meta.yml、目标驱动交付可工作软件

技能详解

编排层

ship-orchestrator

新 ShipKit 统一入口。识别 feature 开发场景，路由到 split/understand/design/build，维护 meta.yml 单一事实源，并执行唯一用户确认门禁。

职责：场景识别、阶段路由、状态推进、门禁执行
场景：quick_start、full_flow、prd_direct、split_first
核心门禁：Design → Build 需用户确认
单一事实源：.docs/feature-*/meta.yml
可选前置

ship-split

把大需求拆成可独立交付的小 feature，生成 splits.yml，标注依赖，并可准备 TAPD/Jira 任务数据。

触发时机：明确要求拆分、5+ 独立功能点、预计超过 3 天
输出：splits.yml + 批量 feature 目录
依赖追踪：自动检查 blocked_by 状态
集成：TAPD/Jira 任务同步框架
核心阶段

ship-understand

加载 spec，解析需求/PRD/原型，必要时调用 grill-me，产出 requirements.md(status: ready)。

目标：把用户输入变成可设计、可验证的需求
输出：requirements.md (功能概述、Domain 模型、AC、非功能需求、约束)
质量标准：至少 1 个 AC，使用 Given/When/Then
grill-me：full_flow 必须，quick_start 条件触发
核心阶段

ship-design

基于 requirements.md 和 spec 生成合一技术设计：API Contract、数据模型、前后端方案，并在 Build 前等待用户确认。

输出：design.md (合一设计文档)
核心章节：AC 覆盖映射、API 契约、数据模型、前后端设计、性能/风险
模板引用：Design Reference Template (项目级或内置)
用户确认：Design → Build 唯一硬门禁
核心阶段

ship-build

用户确认设计后，按 design.md 生成任务清单、实现代码、运行测试、验证 AC，并生成 verification.md。

输出：build-plan.yml + verification.md + 实际代码
任务顺序：数据层 → 后端 → 前端 → 集成
成功标准：all_ac_passed: true + 测试通过
spec 更新：更新 existing-features，按评分沉淀新规范
工具技能

ship-spec

维护 .docs/spec 知识库，按阶段和项目隔离加载规范，并在 Build 完成后更新已有功能索引。

目录：.docs/spec/ (frontend/backend/shared)
加载策略：按阶段加载，多项目隔离
优先级：requirements/AC > 项目 spec > 代码事实 > 模板 > AI 习惯
沉淀评分：复用次数≥3(+40) + 模式稳定(+30) + 跨模块(+20) ≥60
质询助手

ship-grill-me

Understand/Design ready 前发现 blocking 问题，先查仓库和 spec，再一次只问用户一个关键问题。

嵌入位置：ship-understand / ship-design
blocking 判定：缺 AC、冲突、边界缺失、API/数据模型问题
质询流程：先查仓库/spec → 自动修正或提问 → 一次一个问题
触发矩阵：full_flow 必须，quick_start 条件触发

流程说明

  • 实线框：必须阶段，所有 feature 都必须经过
  • 虚线框：可选阶段，根据场景决定是否触发
  • 橙色边框：并行执行的辅助技能 (ship-spec 贯穿全程)
  • 红色高亮：唯一用户确认门禁，Design → Build 必须用户明确同意
  • ship-grill-me：嵌入式质询，在 Understand/Design 中按场景触发

设计原则

简化而不简陋

减少阶段数量，合并文档，但保持质量门禁。旧 ShipKit 的 14 阶段精简为 3 个核心阶段：Understand → Design → Build。

单一事实源 meta.yml

所有 feature 状态、阶段、产物引用都存在 .docs/feature-*/meta.yml。不依赖对话历史或隐式记忆。

目标驱动交付可工作软件

产物必须推进交付。requirements.md 必须有 AC，design.md 必须可实现，verification.md 必须证明所有 AC 通过。

唯一用户确认门禁

Design → Build 是唯一需要用户明确确认的门禁。其他阶段转换由 AI 自检 + validator 验证。

场景驱动的质询

quick_start 跳过多余质询，full_flow 必须质询，prd_direct 条件触发。一次只问一个问题，先查仓库/spec 再问用户。

spec 优先级链

requirements/AC > 项目 spec > 现有代码事实 > 参考模板 > AI 默认习惯。模板只是梤查清单，不替代项目约束。

核心 Validators

validate_requirements.py - 检查 AC 格式、Domain 模型、spec 引用
validate_design.py - 检查 AC 覆盖、API 契约、数据模型、模板引用、错误响应
validate_build.py - 检查 AC 验证、测试结果、产出文件
validate_current_stage.py - 聚合入口，根据 current_stage 调用对应 validator

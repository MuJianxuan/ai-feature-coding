项目总览

这套技能把 skills/ 重构为单一完整推进的 ShipKit feature flow。它不覆盖普通问答或小修 bug；一旦进入 ShipKit feature flow，就按完整流程推进：Understand → Design → Build → Done。

实现根目录：skills/
版本：shipkit-full-flow-v2
设计源：.docs/shipkit-skills-refactor-design.md
单一事实源：.docs/feature-*/meta.yml（跨窗口持久化）
核心 Workflow：full_flow — 单一完整推进，无分叉场景

技能详解

编排层

ship-orchestrator

ShipKit feature flow 的统一入口。负责访谈、创建/恢复 feature 目录、维护 meta.yml、路由阶段，并持久化 Design → Build 用户确认。

入口判断：区分 feature 开发 vs 普通问答/小 bug，只有 feature 进入 flow
固定访谈：确认目录名、读取 project.yml 确认 workspace scope、收集需求来源(source_refs)
目录管理：创建/恢复 .docs/feature-* 目录与 meta.yml，唯一能创建 feature 目录的技能
Build 门禁：唯一硬用户确认门禁，持久化 build_approved_at/by/note 三字段
阻塞恢复：6 种 blocked_reason 处理策略（包括 grill-me pending、validation failed 等）
Handoff：含 feature_dir 路径、下一阶段技能名、workspace mode、projects
可选前置

ship-split

把大需求拆成可独立交付的小 feature 建议，只产出 splits.yml，不直接创建 feature 目录——创建权属于 orchestrator。

触发时机：用户说"拆分"、需求含 5+ 独立功能点、单需求预计超 3 天
粒度控制：单个子需求 ≤ 3 天
输出：splits.yml（id/name/slug/priority/estimated_days/dependencies/blocked_by）
Handoff：输出批量创建 handoff 给 orchestrator，不写 meta.yml、不创建 .docs/feature-*
集成：TAPD/Jira 可导入字段（无凭证不调接口）
核心阶段

ship-understand

把 feature_dir 中登记的需求来源变成可设计、可验证的 requirements.md，作为后续 Design/Build 的唯一需求源。

硬前置：必须接收 feature_dir，meta.yml 存在可读，source_refs 至少 1 个 primary 且可读
探索范围：按 workspace scope 限定既有功能探索
输出：requirements.md（功能概述 / Domain 模型 / AC(Given-When-Then) / NFR / 约束 / 风险）
质量标准：至少 1 个 AC，必须使用 Given/When/Then 格式
Blocking Review：ready 前必须调用 ship-grill-me，无例外
约束：不创建目录、不猜 feature_dir、不做技术方案、不写业务代码
核心阶段

ship-design

基于 ready requirements 选择 Design Reference Template，产出统一 design.md（API 契约 / 数据模型 / 前后端设计 / AC 覆盖映射），完成后 handoff 回 orchestrator。

硬前置：feature_dir 明确、requirements.md status: ready、validate_requirements.py 无 errors
模板选择：7 种内置模板（backend-service / frontend-ui / fullstack-feature / async-task / data-migration / integration / config-change）
输出：design.md（方案模板引用 / AC 覆盖映射 / API 契约 / 数据模型 / 前后端设计 / 性能 / 风险 / 审查记录）
Blocking Review：ready 前必须调用 ship-grill-me 做对抗式设计审查
关键约束：不得把 current_stage 改为 build；不直接启动 ship-build；handoff 回 orchestrator 处理 Build approval
核心阶段

ship-build

把 ready design 变成可工作的代码，成功标准是"可交付"而非"写完代码"——所有 AC 必须被测试/验证覆盖。

硬前置：build_approved_at/by/note 三字段已由 orchestrator 写入；design.md status: ready
输出：build-plan.yml（单任务 ≤ 60min）+ 业务代码 + verification.md（AC→测试逐项映射）
实现顺序：数据层 → 后端 → 前端 → 集成
测试要求：每任务运行相关测试 + 项目真实 test/lint/typecheck
成功标准：all_ac_passed: true + 测试通过 + validate_build.py 无 errors
Spec 更新：Build 完成后更新 existing-features 或提出沉淀建议
工具技能

ship-spec

维护 .docs/spec 知识库，按 meta.yml workspace/projects 隔离加载规范，在 Build 完成后更新 existing-features 索引。

阶段分类加载：Understand(existing-features/domain) → Design(API/frontend/backend/database) → Build(coding/test/deployment)
项目隔离：永远加载 _shared + 仅 meta.yml.projects 指定项目
优先级链：requirements/AC > 项目 spec > 现有代码事实 > 参考模板 > AI 默认习惯
沉淀评分：复用次数≥3(+40) + 模式稳定(+30) + 跨模块(+20) ≥ 60 分才沉淀
关键边界：spec ≠ 参考模板（模板是 checklist，spec 是项目约束）；spec ≠ source_refs（PRD 不塞 spec）
Blocking Review

ship-grill-me

Understand/Design ready 前必须执行的 blocking review：先查证据自答，再一次只问用户一个关键阻塞问题——无例外。

触发位置：Understand ready 前 + Design ready 前，必须执行，无场景豁免
审查流程：扫描 blocking 问题 → 从仓库/spec/source_refs 查证据 → 能自答就自动修正 → 不能则只问用户 1 个
提问格式：阻塞说明 + 已检查证据 + 推荐答案 + 影响分析
Understand blocking：无可测 AC、AC 缺 GWT、需求冲突、引用未定义功能、边界缺失影响 DB/API
Design blocking：API 缺 Req/Res/Error、数据模型冲突、AC 未覆盖、模板必填缺失、含 TBD/待补充
输出：result: ready|blocked + resolved/pending/warnings 列表

流程说明

  • 粗实线框（orchestrator）：统一入口，管理 feature 生命周期和阶段路由
  • 实线框：核心阶段技能，所有 feature 必须经过
  • 虚线框：可选阶段，根据需求规模决定是否触发
  • 橙色边框（ship-spec）：按阶段分类加载，贯穿 Understand/Design/Build
  • 深蓝边框（ship-grill-me）：必须执行的 blocking review，Understand/Design 阶段无例外
  • 红色高亮：唯一用户确认门禁，由 orchestrator 持久化 Build approval 三字段
  • 灰色标注：阶段间 handoff 携带的关键信息和 validator 检查点

核心机制

Workspace Scope 机制

.docs/ship/project.yml 定义仓库默认配置，meta.yml 定义 feature 实际范围。所有阶段技能按此隔离探索/加载/实现。

workspace_mode：single_project | project_group
projects 列表：限定 spec 加载和代码探索的项目范围
隔离加载：永远加载 _shared + 仅指定项目的 spec
Design Reference Template

轻量参考模板机制：orchestrator 记录用户意图，ship-design 选择并引用模板，模板只是 checklist，不替代项目约束。

7 种内置模板：backend-service / frontend-ui / fullstack-feature / async-task / data-migration / integration / config-change
优先级：用户指定 > 项目级模板 > 内置模板
记录位置：meta.yml 的 design_template_ref + design_template_reason
Source Refs 来源追踪

需求来源在 orchestrator 阶段登记到 meta.yml.source_refs，贯穿全流程作为需求可追溯依据。

登记内容：id / type(prd/ui-ux/link/issue) / title / path_or_url / role(primary/secondary) / status
Understand 使用：读取来源材料提取需求
Grill-me 使用：查证据自答阻塞问题
Build Approval 持久化

Design → Build 是唯一硬门禁。用户确认后由 orchestrator 写入三个字段到 meta.yml，ship-build 启动时校验。

build_approved_at：确认时间戳
build_approved_by：确认人标识
build_approval_note：确认备注（如"按设计执行"或附加要求）
Spec 沉淀评分

Build 完成后自动评估是否值得将新模式沉淀为 spec，避免无意义膨胀。

评分标准：复用次数≥3(+40) + 模式稳定(+30) + 跨模块(+20)
沉淀门槛：≥ 60 分才建议沉淀为 spec
更新范围：existing-features.md（表/API/页面/关键测试）
Validator 验证体系

结构和覆盖性检查，贯穿每个阶段。不替代项目自身测试。

validate_requirements.py：AC 格式、Domain 模型、spec 引用、source_refs
validate_design.py：AC 覆盖、API 契约、数据模型、模板引用、错误响应
validate_build.py：AC 验证、测试结果、产出文件
validate_current_stage.py：聚合入口，根据 current_stage 调用对应 validator

设计原则

单一完整推进 full_flow

不分叉、不跳阶段。每个 feature 都走 Understand → Design → Build → Done 完整链路，阶段间通过 validator 保证质量。

meta.yml 跨窗口单一事实源

所有 feature 状态、阶段、产物引用、Build approval 都存在 .docs/feature-*/meta.yml。不依赖对话历史或隐式记忆。恢复 feature 必须读 meta.yml。

阶段技能必须显式接收 feature_dir

所有核心阶段技能（Understand/Design/Build）都必须通过 handoff 显式接收 feature_dir。缺 feature_dir 直接停止，不猜测。

唯一用户确认门禁

Design → Build 是唯一需要用户明确确认的门禁，由 orchestrator 持久化 build_approved_* 三字段。其他阶段转换由 AI 自检 + validator 验证。

只有 orchestrator 创建 feature 目录

ship-split 只产出建议，阶段技能只操作已有目录。feature 目录和初始 meta.yml 的创建权唯一属于 orchestrator，保证目录结构一致性。

workspace scope 限定范围

project.yml 定义默认配置，meta.yml.projects 定义 feature 实际范围。所有技能按此隔离：spec 加载、代码探索、实现范围都不超出 projects。

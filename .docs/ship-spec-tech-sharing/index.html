ship-spec 核心设计理念

💡 按需加载

不是百科全书，按阶段（Understand/Design/Build）和项目加载所需规范

🔍 动态索引

零合并冲突：扫描 frontmatter 动态生成索引，无需维护中心文件

🎯 质量门禁

60分评分标准：低分规范不入库，防止知识库污染

设计原则

  • 单一数据源：meta.yml + frontmatter 是唯一事实
  • 工具驱动：ship-spec CLI 自动化规范管理
  • 项目优先：项目规范 > 通用规范 > AI 默认习惯

规范采集标准：60 分评分机制

评分标准（≥ 60 分才建议新增）

✅ 复用次数 ≥ 3：+40 分

多次复用才值得沉淀

✅ 模式稳定：+30 分

6 个月内不会变化

✅ 跨模块使用：+20 分

被多个模块/团队使用

✅ 中高复杂度：+10 分

有一定复杂度，非显而易见

💡 原则：低分就别污染知识库。现实里最烂的规范库就是没人敢删、没人会读。

对抗式审查清单

✅ 必要性检查

  • 复用次数 ≥ 3 次？
  • 模式稳定（未来 6 个月）？
  • 跨模块使用？
  • 有一定复杂度？

✅ 完整性检查

  • frontmatter 完整
  • 有适用场景
  • 有核心约束
  • 有具体示例

✅ 简洁性检查

  • 规范内容 ≤ 200 行
  • 只写约束，不写实现
  • 与现有规范无重复
  • 每句话都有价值

✅ 安全性检查

  • 无敏感信息（密码/token/密钥）
  • 无生产数据
  • 只描述来源

审查结论：

✅ APPROVED - 所有检查通过 | ❌ REJECTED - 必须修复 | ⚠️ NEEDS IMPROVEMENT - 建议改进

ship-spec CLI 工具

安装

npm install -g @shipkit/spec-cli
ship-spec -h

基本命令

# 初始化目录
ship-spec init --mode multi --projects web,api

# 创建规范
ship-spec create rest-api -t backend -d "REST API 设计规范"

# 列出规范（动态扫描 frontmatter）
ship-spec list --stage design --format json

# 加载规范
ship-spec load rest-api --content-only

# 验证规范
ship-spec validate --all --strict

# REST API 设计规范

## 适用场景
- 设计新的 REST API
- 审查现有 API

## 核心约束
### 必须做
- 使用 RESTful 资源命名
- 统一错误格式
- 版本化（/api/v1/）

### 禁止做
- 不在 URL 中使用动词
- 不返回裸数据

## 示例
✅ GET /api/v1/users/123
❌ GET /api/v1/getUser?id=123

## 检查点
- [ ] URL 使用名词复数
- [ ] 错误响应包含 code/message
- [ ] 有版本号

// ShipKit product architecture

把 AI Coding
产品化交付

围绕 skills/ship-spec、ShipKit 技能流和 tools/ship-spec CLI 的 10 分钟技术分享

Product View 10 min Cyberpunk Mode
ShipKit feature flow · spec CLI · product architecture

今天这 10 分钟，我不从代码细节开始讲，而是从产品视角讲 ShipKit 为什么存在。

我们面对的不是“让 AI 写一段代码”这么简单的问题，而是让 AI 持续、可控、可复盘地交付 feature。这里面最难的不是生成能力，而是流程、边界、证据和组织记忆。

所以这份分享会围绕三层展开：第一层是 skills/ 下的 ShipKit 技能流；第二层是 ship-spec 这套规范知识库方法；第三层是 tools/ship-spec CLI 怎么把规范变成可查询、可加载、可验证的工具。

最后我会给一个使用路径：团队怎么从“口头约定”走到“可执行规范”。

// 01 problem

AI Coding 的真实瓶颈
不是会不会写代码

01

需求失真

对话里确认过，不代表跨窗口、跨阶段还能保持一致。

02

规范漂移

团队约定散在脑子里、旧文档里、代码习惯里，AI 很容易按默认习惯发挥。

03

完成幻觉

“写完了”和“可交付”之间，缺少 AC、测试和验证证据。

产品目标：把 AI 从“聪明的执行者”约束成“可审计的交付系统”。

先看问题。AI Coding 的瓶颈，已经不只是“模型会不会写代码”。模型当然会写，但是团队真正怕的是三件事。

第一是需求失真。很多需求在一个窗口里讲清楚了，换个阶段、换个上下文就开始丢。第二是规范漂移。项目里有很多默认规则，比如 API 风格、错误码、组件模式、测试边界，但这些东西常常没有一个机器能稳定加载的入口。第三是完成幻觉。AI 很容易说“完成了”，但是 AC 有没有覆盖，测试有没有跑，验证证据在哪里，不一定清楚。

所以 ShipKit 的产品目标不是让 AI 更自由，而是相反：给它一个可审计的交付系统，把聪明约束在正确轨道里。

// 02 design principles

ShipKit 的设计理念：
把不确定性分层封装

Single Source of Truth

meta.yml 记录阶段、scope、source_refs、approval，不靠对话记忆。

Stage Gates

Understand → Design → Build，每一步都有产物、validator 和明确 handoff。

Evidence First

先查仓库、source_refs、spec，再问用户；问题必须带证据和影响。

Spec as Product Memory

可复用、稳定、跨模块的约束才沉淀；低价值内容不污染知识库。

ShipKit 的核心理念，我会概括成一句话：把不确定性分层封装。

第一层是事实源。仓库里的说明写得很明确，feature 状态以 .docs/feature-*/meta.yml 为单一事实源。它记录当前阶段、workspace scope、需求来源、spec 引用和 Build approval。这样跨窗口恢复时，不需要靠记忆。

第二层是阶段门禁。Understand 产出 requirements.md，Design 产出 design.md，Build 产出 verification.md。每一步都有 validator，不是“感觉差不多”。

第三层是证据优先。ship-grill-me 的规则是先查仓库、spec 和来源材料，找不到才问用户，而且一次只问一个阻塞问题。

最后一层，就是 ship-spec：把规范当成产品记忆，而不是没人读的文档仓库。

// 03 skill flow

技能不是列表，是一条交付流水线

optional ship-split

大需求拆分，只产出 splits.yml

entry ship-orchestrator

创建/恢复 feature_dir，维护 meta.yml

stage 1 ship-understand

来源 + spec + 代码事实 → requirements.md

stage 2 ship-design

模板 + spec + AC 映射 → design.md

stage 3 ship-build

实现 + 测试 + AC 验证 → verification.md

embedded: ship-grill-me blocks bad requirements/design before ready

接下来把技能流摊开看。这里最重要的是：skills 不是工具列表，而是一条交付流水线。

ship-split 是可选前置，用来处理大需求拆分。它只给建议和 splits.yml，不越权创建 feature 目录。真正的入口是 ship-orchestrator，它负责创建或恢复 .docs/feature-*，并维护 meta.yml。

然后进入三个核心阶段。ship-understand 把来源材料、spec 和代码事实变成 requirements.md。ship-design 选择 Design Reference Template，输出前后端合一的 design.md。ship-build 在 Build approval 存在后才实现，并写 verification.md。

中间还有一个嵌入式质询助手 ship-grill-me，它的作用是把会返工的问题挡在 ready 之前。

// 04 ship-spec skill

ship-spec：
把“团队经验”变成可加载约束

.docs/spec/
├── frontend/
├── backend/
└── shared/
    ├── tech-stack.md
    ├── existing-features.md
    └── naming-conventions.md
沉淀门槛

复用 ≥ 3、模式稳定、跨模块、中高复杂度，评分 ≥ 60 才建议新增。

质量标准

不超过 200 行，聚焦约束，有正反例，有检查点。

边界感

spec 是项目约束；Design Reference Template 是方案护栏，不能互相替代。

现在进入重点：ship-spec 到底解决什么问题。

它管理的是 .docs/spec/ 知识库。这里放的不是普通文档，而是项目约束。比如技术栈、已有功能、命名规范、API 标准、前端组件模式。这些东西一旦能被 CLI 加载，就能在 Understand、Design、Build 每个阶段进入上下文。

但它不是鼓励大家疯狂写规范。ship-spec 反而有很强的产品克制：有一套 60 分沉淀标准。复用次数、稳定性、跨模块使用、复杂度都要考虑。低分内容不要污染知识库。

质量上也很硬：规范要简明、可执行、可验证，有正确和错误示例，有检查点。还有一个边界：spec 是项目约束，Design Reference Template 是方案护栏，两者不能混用。

// 05 ship-spec CLI

CLI 的产品价值：
让规范进入执行路径

ship-spec init

初始化 single / multi workspace 的 spec 目录。

ship-spec create

按 spec_id、type、project、stages 创建规范草稿。

ship-spec list

扫描 `.docs/spec` 下 Markdown frontmatter，按 project / stage 过滤。

ship-spec load

按 spec_id 输出 JSON / YAML / content-only，供技能阶段加载。

ship-spec validate

验证规范文件存在性和 frontmatter 一致性。

ship-spec migrate

兼容旧配置格式，把历史项目迁到新结构。

实现备注：当前 `list` 已动态扫描 frontmatter；`validate` 仍依赖 `INDEX.md`，这是迁移中的边界。

如果只有 skill，没有工具，规范还是很容易停留在“写过但没进流程”。所以 tools/ship-spec 的产品价值，就是让规范进入执行路径。

CLI 提供六类命令。init 初始化目录，支持 single 和 multi workspace。create 创建规范草稿。list 列出规范，而且实现里会递归扫描 .docs/spec 下的 Markdown frontmatter，再按 project 和 stage 过滤。load 把规范按 JSON、YAML 或正文输出，方便阶段技能加载。validate 做格式校验。migrate 处理旧配置兼容。

这里我也刻意讲一个真实边界：README 和技能目标都强调动态扫描，但当前实现里 validate 仍依赖 INDEX.md。所以这不是完美终态，而是一个正在产品化的工具链。

这个透明度很重要，因为产品设计不是包装完美，而是明确能力边界。

// 06 operating model

落地方式：
从一次 feature 交付到团队知识循环

1 交付前

orchestrator 确认 workspace scope 和 source_refs。

2 交付中

各阶段按 project/stage 加载 spec，grill-me 阻断关键缺口。

3 交付后

build 写 verification，更新 existing-features，评估是否新增规范。

4 下一次

新 feature 直接复用已沉淀约束，减少重复访谈和返工。

Product KPI少问重复问题
Engineering KPI少做无证据改动
Quality KPIAC 可追踪验证

从产品落地看，ShipKit 不是一次性写文档，而是一个团队知识循环。

交付前，orchestrator 先确认 workspace scope 和需求来源。这个动作保证后续探索范围不会失控，也保证需求有可追溯入口。

交付中，各阶段按 project 和 stage 加载 spec。比如 Understand 更关心已有功能和业务词汇，Design 更关心 API、数据模型、前端模式，Build 更关心编码和测试约束。ship-grill-me 会把影响实现的缺口挡住。

交付后，Build 阶段不是停在“代码写完”，而是写 verification.md，逐项映射 AC；同时更新 existing-features，并按 60 分规则判断是否有新规范值得沉淀。

下一次 feature 再来，团队就少问重复问题，AI 也少做无证据改动。

// 07 takeaways

最后带走三句话

01 流程不是束缚，是降噪器。

AI 越强，越需要清晰边界和阶段门禁。

02 spec 不是文档，是执行上下文。

能被 list / load / validate，才算进入交付系统。

03 可交付不是写完，是证据闭环。

requirements、design、verification 串起来，才有产品级可信度。

ShipKit = workflow + memory + gates + proof
Q&A · @skills/ship-spec · @tools/ship-spec

最后收一下，今天我希望大家带走三句话。

第一，流程不是束缚，是降噪器。AI 越强，越容易“顺手多做一点”，所以更需要 scope、stage gate 和 handoff。

第二，spec 不是文档，是执行上下文。如果它只能给人读，那价值有限；如果它能被 list、load、validate 进入技能流程，它就会影响每一次交付。

第三，可交付不是写完，是证据闭环。ShipKit 用 requirements.md 固定需求，用 design.md 固定方案，用 verification.md 固定验证，再用 meta.yml 把阶段状态串起来。

所以我会把 ShipKit 概括成四个词：workflow、memory、gates、proof。它的价值不是替你写更多代码，而是让团队更敢把 feature 交给 AI 推进。谢谢，接下来可以讨论。

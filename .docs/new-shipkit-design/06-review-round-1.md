# 第一轮审查：问题识别

## 审查日期：2026-06-09

### 已完成部分
✅ 核心 4 阶段定义（Split, Understand, Design, Build）
✅ spec 工具技能设计
✅ grill-me 嵌入机制
✅ 状态管理极简化（meta.yml 8 字段）
✅ 场景模式定义（4 种）
✅ 多项目支持

### 发现的问题

#### 1. 缺少编排层（Orchestrator）
**问题**：没有定义统一入口和路由机制
- 用户如何启动流程？
- 如何判断从哪个阶段开始？
- 如何恢复中断的工作？

**解决**：需要设计 ship-orchestrator 简化版

#### 2. Split 技能定义不完整
**问题**：
- TAPD API 集成细节缺失
- 拆分后如何批量创建 feature？
- 依赖关系如何在后续阶段体现？

**解决**：补充批量 feature 创建流程

#### 3. grill-me 嵌入时机不明确
**问题**：
- 何时触发 grill-me？
- 如何判断问题是 blocking 还是 non-blocking？
- quick_start 模式真的能跳过吗？

**解决**：细化 grill-me 触发规则

#### 4. Design → Build 的确认机制不清晰
**问题**：
- 只需要 yes/no 是否太简单？
- 用户可能想修改设计方案
- 如何处理"部分确认，部分修改"？

**解决**：补充确认流程和修改循环

#### 5. Build 阶段缺少实施细节
**问题**：
- 如何生成任务清单？（算法不明确）
- 任务粒度如何控制？
- 前后端并行还是串行？
- 失败重试机制？

**解决**：补充实施算法和错误处理

#### 6. spec 自动更新机制不完整
**问题**：
- 谁来判断"has_reusable_patterns"？
- 如何避免规范冗余？
- 规范版本管理？

**解决**：细化规范沉淀规则

#### 7. 缺少 Validator 设计
**问题**：
- 说要 3 个 validator，但没定义具体逻辑
- 如何验证 requirements 和 design 的质量？

**解决**：补充 validator 逻辑

#### 8. 缺少错误处理和阻塞恢复
**问题**：
- 如果 grill-me 发现阻塞问题怎么办？
- 如果 Build 阶段测试失败怎么办？
- 如何回退到上一阶段？

**解决**：补充状态回退和错误处理

#### 9. 文档结构不完整
**问题**：
- feature 目录结构没定义
- 产物文件命名规范不明确

**解决**：补充目录结构定义

#### 10. 缺少实际使用示例
**问题**：
- 理论设计没有实际场景演练
- 不确定是否真的简化了

**解决**：需要写 2-3 个完整的使用案例

# 技术选型 (Tech Selection)

## Overview

本阶段基于 tech-research.md 的调研结果，做出明确的技术栈和架构选型决策。核心原则：**每个决策都必须有备选方案、有不选的理由、有可追溯的 ADR（Architecture Decision Record）**。

产出物：`tech-selection.md`

## When to Use

- tech-research.md 已完成（stage_status: ready, evidence_complete: true）
- 项目启动前需要确定技术栈
- 现有项目引入新技术或重大架构调整
- 需要在多个候选方案中做出决策

## When NOT to Use

- tech-research.md 未完成或信息不全
- 仅是技术学习探索，不涉及实际项目决策
- 选型已由组织/平台强制规定（此时只需文档化已有决策）

## Process

```
1. 读取 tech-research.md → 提取候选方案
   verify: 每个技术领域至少有 1-3 个候选
2. 读取 requirements.md → 提取约束和非功能需求
   verify: 约束清单完整（团队/预算/时间/基础设施）
3. 定义选型评估维度和权重
   verify: 维度对齐非功能需求，权重有明确依据
4. 逐项做出决策，撰写 ADR
   verify: 每个 ADR 包含上下文/决策/理由/后果/备选方案
5. 验证选型与非功能需求的兼容性
   verify: 逐条对照 requirements.md 中的非功能需求，无遗漏
6. 确定架构模式
   verify: 模式与团队规模、流量预期、迭代节奏匹配
7. 如为新项目，提供脚手架初始化方案
   verify: 给出官方推荐的初始化命令，可直接执行
8. 写入 tech-selection.md
   verify: frontmatter 完整，决策表和 ADR 齐全，无 TODO 遗留
```

### 决策流程原则

- **证据驱动**：每个决策必须引用 tech-research.md 中的具体数据
- **显式权衡**：不存在"完美方案"，必须说明取舍
- **可逆性评估**：标注每个决策的变更成本（高/中/低）
- **时间约束**：选型不应无限期，设定决策截止时间

## ADR Format (架构决策记录格式)

每个独立决策对应一个 ADR：

```markdown
### ADR-XXX: [决策标题]

**状态**: proposed / accepted / deprecated / superseded by ADR-YYY

**上下文 (Context)**
为什么需要做这个决策？背景是什么？面临什么问题？

**决策 (Decision)**
选择了什么方案。一句话说清楚。

**理由 (Rationale)**
为什么选它？对比了哪些备选方案？关键的取舍在哪里？
引用 tech-research.md 中的具体数据/来源。

**后果 (Consequences)**
- 正面：带来的好处
- 负面：引入的成本和复杂度
- 风险：潜在问题及缓解措施

**备选方案 (Alternatives Considered)**
- 方案 A：[名称] — 不选的理由
- 方案 B：[名称] — 不选的理由
```

## Selection Criteria (选型评估维度)

根据 requirements.md 中的非功能需求，从以下维度评估：

| 维度 | 评估问题 | 权重示例 |
|------|----------|----------|
| 功能匹配度 | 是否满足核心功能需求？ | 必须 |
| 性能 | 能否满足性能 SLA？ | 高 |
| 可扩展性 | 是否支持未来增长？ | 中-高 |
| 团队熟悉度 | 团队当前技能匹配度？ | 高 |
| 学习成本 | 上手时间和培训需求？ | 中 |
| 生态成熟度 | 社区/工具/集成完善度？ | 中-高 |
| 长期维护 | 项目活跃度、商业支持？ | 高 |
| 总体成本 | License、基础设施、人力 | 中-高 |
| 安全合规 | 是否满足安全/合规要求？ | 必须 |
| 迁移成本 | 与现有系统的集成难度？ | 中 |
| 可逆性 | 后期更换成本？ | 中 |
| 可观测性 | 监控/日志/追踪生态？ | 中 |

权重应根据项目实际情况调整，并在文档中说明。

### 加权评分模板

对关键决策（如核心框架），可使用加权评分：

```markdown
| 维度 | 权重 | 方案 A 得分 | 方案 A 加权 | 方案 B 得分 | 方案 B 加权 |
|------|------|-------------|-------------|-------------|-------------|
| 性能 | 30% | 8 | 2.4 | 9 | 2.7 |
| 团队熟悉度 | 25% | 9 | 2.25 | 5 | 1.25 |
| 生态 | 20% | 7 | 1.4 | 9 | 1.8 |
| 学习成本 | 15% | 8 | 1.2 | 6 | 0.9 |
| 长期维护 | 10% | 8 | 0.8 | 8 | 0.8 |
| **合计** | 100% | — | **8.05** | — | **7.45** |
```

注意：评分仅辅助决策，不应机械化使用。最终决策必须结合定性判断。

### 核心章节

1. **选型摘要** — 一句话结论（"采用 X + Y + Z 技术栈，基于 A 架构模式"）

2. **技术栈决策表**

   | 层级 | 选型 | 版本 | 理由 | 备选方案 |
   |------|------|------|------|----------|
   | 前端框架 | ... | ... | ... | ... |
   | 后端框架 | ... | ... | ... | ... |
   | 数据库 | ... | ... | ... | ... |
   | 缓存 | ... | ... | ... | ... |
   | 消息队列 | ... | ... | ... | ... |
   | 部署环境 | ... | ... | ... | ... |

3. **ADR 列表** — 每条决策一个 ADR（按上述格式）

4. **架构模式选择** — 单体 / 微服务 / 模块化单体 / Serverless / ...
   - 选择的模式
   - 理由（基于团队规模、流量预期、迭代节奏）
   - 演进路径（未来如何调整）

5. **关键约束** — 影响选型的边界条件
   - 团队技能现状
   - 预算限制
   - 时间窗口
   - 基础设施约束（云厂商/合规要求）

6. **与 requirements.md 非功能需求的对齐验证**

   | 非功能需求 | 选型如何满足 | 风险/缺口 |
   |------------|--------------|-----------|
   | 性能 SLA | ... | ... |
   | 可用性 | ... | ... |
   | 安全 | ... | ... |
   | 可扩展性 | ... | ... |

7. **项目初始化方案**（仅新项目）
   - 脚手架命令（官方推荐方案优先）
   - 初始目录结构
   - 关键配置项
   - 依赖管理方式

### stage_status 流转规则

- `draft`：选型进行中，仍有未决项
- `ready`：所有决策已完成，ADR 齐备，可进入设计阶段

### evidence_complete 判定标准

- 每个决策都有 ADR
- 每个 ADR 都列出了至少 1 个备选方案
- 所有非功能需求都有对齐验证
- 新项目已给出脚手架初始化方案

## Anti-Rationalizations

1. **"我们一直用 X，继续用 X 就行"** — 不行。即使保留 X 也要写 ADR，明确"基于团队熟悉度选择保持现状"，并验证 X 是否仍满足当前需求。
2. **"这个技术明显最好，不需要备选"** — 必须列出至少一个备选并说明不选的理由。"明显最好"通常意味着调研不充分。
3. **"先选了再说，后面可以换"** — 选型是高成本变更。必须验证与非功能需求的兼容性，避免后期推倒重来。
4. **"团队不熟悉但技术先进，先用着学"** — 学习成本必须体现在 ADR 的"后果"中，并评估对项目时间线的影响。
5. **"非功能需求太多，挑重点验证就行"** — 漏掉的需求往往是项目后期最大的坑。逐条对齐验证，不能跳过。

## Red Flags

- 决策表中只有"选型"没有"备选方案"
- ADR 的"理由"是"业界主流"或"听说很好"，没有具体证据
- 选型与 tech-research.md 的调研结论矛盾且无说明
- 非功能需求对齐表中存在大量"待验证"项
- 新项目未给出脚手架初始化方案，让团队从零搭建
- 架构模式选择与团队规模严重不匹配（小团队微服务 / 大流量纯单体）

## Verification

完成 tech-selection.md 后，执行以下检查：

```
□ 技术栈决策表是否覆盖所有层级？
□ 每个决策是否有对应的 ADR？
□ 每个 ADR 是否包含上下文/决策/理由/后果/备选方案？
□ 是否引用了 tech-research.md 中的具体证据？
□ 非功能需求对齐表是否逐条验证？
□ 架构模式是否与团队规模和需求匹配？
□ 新项目是否给出官方推荐的脚手架命令？
□ 是否有未解决的高风险项？
□ evidence_complete 字段是否准确？
```

全部通过后，将 `stage_status` 设为 `ready`，可进入下一阶段（API 契约设计）。

# Coding Spec Management

## 目标

帮助用户将编码经验、架构决策和技术约束沉淀为简洁可执行的规范文档，维护在 `.docs/spec/` 目录下。

## Activation policy

本 skill 在以下情况下使用：

1. 用户明确写出 `coding-spec-management`，或明确要求"添加规范 / 新增规范 / 更新规范 / 管理规范"。
2. 用户提供了规范相关的输入（代码片段、文档、提示词）并要求沉淀为规范。

不满足时：

- 不得自动触发。
- 不得在其他 coding-* skill 执行过程中自动创建规范。

## 前置检查

1. 确认 `.docs/spec/INDEX.md` 存在。如果不存在，从本 skill 的 `assets/INDEX.md` 复制到 `.docs/spec/INDEX.md`（同时创建 `.docs/spec/` 目录）。
2. 确认 `.docs/spec/coding/` 目录存在。如果不存在，创建该目录。

## 输入来源

规范可以基于以下任一来源生成：

- **代码现状**：用户指定文件或模块，AI 从中提炼模式和约束。
- **文档**：用户提供的技术文档、最佳实践文章或内部 wiki。
- **提示词**：用户口头描述的原则或经验。

## 工作流

### Step 1：理解意图

询问或推断：
- 规范主题是什么（如 redis、mq、日志、异常处理）。
- 规范的适用场景和边界。
- 输入来源类型。

如果用户已经给出足够信息，跳过询问直接进入下一步。

### Step 2：判断新建 vs 补充

读取 `.docs/spec/INDEX.md`：

- 如果索引中已有同主题文档：读取该文档，判断用户输入是补充已有内容还是需要新建独立文档。
  - 内容属于同一主题的细化或补充 → 更新已有文档。
  - 内容是独立子主题或完全不同的关注点 → 新建文档。
- 如果索引中无同主题文档 → 新建文档。

### Step 3：提炼规范

从输入来源中提炼：
- 3-5 条核心原则（思想层面，回答"为什么"）。
- 若干条具体规约（DO / DON'T，可执行，回答"怎么做"）。
- 必要时附一个正例或反例。

原则：大道至简。只写最重要的、容易犯错的、有争议需要统一的。不写显而易见的常识。

### Step 4：写入文档

新建文档时：
- 文件名：`<kebab-case-topic>.md`，放在 `.docs/spec/coding/` 下。
- 从本 skill 的 `assets/spec-template.md` 复制模板，填充内容。
- 更新 `.docs/spec/INDEX.md` 索引表格，新增一行。
- 更新 `INDEX.md` 的 `updated_at`。

补充已有文档时：
- 在对应 section 追加内容。
- 更新文档 frontmatter 的 `updated_at`。
- 如有必要更新 `INDEX.md` 中该文档的描述或更新时间。

### Step 5：确认输出

向用户展示：
- 新增或修改的规范内容摘要。
- 索引变更（如有）。
- 建议用户 review 后确认。

## 规范文档模板

```markdown
---
spec: <kebab-case-name>
scope: coding
updated_at: "<ISO 8601 + timezone>"
---

# <主题> 使用规范

## 原则

- （3-5 条核心原则，一句话一条）

## 规约

- DO: ...
- DON'T: ...

## 示例

（可选。只在规约不够直观时给一个正例或反例片段）
```

## INDEX.md 格式

```markdown
## coding/

| 文件 | 主题 | 适用场景 | 更新时间 |
| --- | --- | --- | --- |
| redis.md | Redis 使用规范 | 缓存、分布式锁、会话 | 2026-05-11 |
```

## Safety policy

- 不删除已有规范文档，除非用户明确许可。
- 不修改规范文档中用户已确认的内容，除非用户要求更新。
- 文档用中文写，保留 technical English names。

## 输出

完成后输出：
- 操作类型（新建 / 补充）。
- 文件路径。
- 规范内容摘要。
- INDEX.md 是否更新。

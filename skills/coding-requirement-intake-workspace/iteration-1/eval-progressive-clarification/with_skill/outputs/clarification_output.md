**Q1.1 [决策] 导出范围的边界**
discovery 中字段清单是"订单主表 + 订单项的合并视图"还是"仅订单主表"？请选：
- 选项 A：**单 Sheet 扁平化**（一行 = 一个订单项，订单字段冗余重复）。便于数据分析透视，文件行数 ≈ 订单数 × 平均订单项数。
- 选项 B：**单 Sheet 订单维度**（一行 = 一个订单，商品名 / 数量按"商品 A x2; 商品 B x1"拼接成一个单元格）。文件更紧凑，但难以做分项分析。
- 选项 C：**双 Sheet**（Sheet1 订单维度，Sheet2 订单项维度，通过订单号关联）。
- Trade-off：A 通用但文件大；B 紧凑但分析不便；C 结构清晰但实现成本最高。

**Q1.2 [决策] 触发导出的入口与同步/异步形态**
- 选项 A：**全异步**——用户提交筛选后立即返回任务 ID，结果在"我的导出任务"列表查看 / 下载。统一体验，几条几万条都一样。
- 选项 B：**条件分流**——预估行数 ≤ 1000 同步直接下载，> 1000 异步排队。体验更"小数据快"，但需要预估查询成本。
- Trade-off：A 简单一致，B 体验更好但要先跑 count 查询。

## 第二层 · 用户路径与业务流程细化（在第一层确认后开启）

> 以下内容**预生成**，等待第一层结论锁定后逐条与用户回放确认。每条路径独立编号，便于用户精确指出修改点。

### 主路径 P1：发起导出 → 下载

| 步骤 | 触发条件 | 角色 | 系统行为 | 预期结果 |
|------|----------|------|----------|----------|
| 1 | 用户进入"订单导出"菜单 | 运营主管 / 财务 | 校验角色权限；展示筛选表单 | 表单加载完成 |
| 2 | 用户填写"下单时间范围 + 订单状态"并点击"导出" | 同上 | 校验时间跨度 ≤ N 天；预估命中行数 | 通过校验后创建异步任务并返回 task_id |
| 3 | 用户跳转 / 切换到"我的导出任务"列表 | 同上 | 列出该用户最近 30 天任务，状态：等待中 / 处理中 / 成功 / 失败 / 已过期 | 用户能看到刚提交的任务 |
| 4 | 任务成功后，用户点击下载 | 同上 | 校验文件未过期、属于本人；返回带签名的下载 URL | Excel 落地 |

### 关键分支与异常路径

- **B1**：筛选结果为 0 行 → 直接同步提示"无数据"，不创建任务。
- **B2**：筛选结果超过单次上限 → 提示用户缩小时间范围；不创建任务。
- **B3**：异步任务执行失败（DB 超时 / 文件写入失败）→ 任务标记为失败，列表展示失败原因摘要；支持"重试"按钮，重试上限 3 次。
- **B4**：用户跨账号访问他人任务 / 文件 → 403。
- **B5**：文件超过保留期（建议 7 天）→ 状态变为"已过期"，下载按钮置灰。

## 第三层 · 边缘场景与非功能约束（在第二层确认后开启）

### 边缘场景候选清单（我主动列出，等待用户取舍）

| ID | 场景 | 我的建议处理 | 状态 |
|----|------|--------------|------|
| E1 | 同一用户短时间内重复提交相同筛选 | 任务级幂等：相同筛选 + 同一用户 + 5 分钟内 → 复用上一条 task_id | ASSUMED |
| E2 | 任务执行中订单数据被更新（如状态变化） | 以"任务创建时间点"为快照基准，不做强一致；导出说明中标注数据时间 | ASSUMED |
| E3 | 大量并发任务挤占数据库 | 全局并发度 ≤ 3 个导出任务；超出排队 | ASSUMED |
| E4 | 文件含敏感字段（收货地址、手机号） | 默认明文导出，下载时记录审计日志（who / when / 筛选条件 / task_id）| ASSUMED |
| E5 | 导出过程中用户登出 | 不影响后台任务执行；用户重新登录后仍可看到任务结果 | ASSUMED |
| E6 | 任务长时间无人下载 | 文件按 Q2.3 过期清理；任务记录保留 30 天 | ASSUMED |

### 非功能要求草案

- **性能**：5 万行导出端到端 ≤ 5 分钟；同步提交接口 P95 ≤ 500ms。
- **安全**：下载 URL 必须签名 + 有效期 ≤ 10 分钟 + 一次性。审计日志含操作人 / IP / 筛选条件 / 文件 hash。
- **兼容性**：Excel 文件兼容 Office 2016+；不强求 WPS。
- **可观测性**：每个任务全链路 traceId；失败需上报告警（量级阈值待定）。
- **合规**：数据保留与日志保留遵循公司既有数据治理策略（无新增个人数据收集）。

### 待用户决策 / 确认的问题（共 5 个）

**Q3.1 [决策] 敏感字段是否脱敏**
- 选项 A：明文导出（与现有 DB 查询一致），仅靠审计日志兜底。
- 选项 B：手机号 / 收货地址在 Excel 中部分掩码（需明确"部分"规则）。
- 选项 C：明文导出，但要求二次确认（如短信验证 / 二次密码）。
- Trade-off：A 最快上线；B 影响财务对账（地址可能要全文）；C 体验差但合规最稳。

**Q3.2 [决策] 全局并发与排队策略**
- 选项 A：全局 3 路并发，超出 FIFO 排队，最多排队 20 个，超出则拒绝并提示。
- 选项 B：按用户限流（每人同时最多 1 个 + 全局 5 个）。
- 选项 C：不做全局限流，依赖 DB 自身保护。
- Trade-off：A 稳但月末高峰排队久；B 公平；C 实现简单但风险高。

**Q3.3 [确认] 审计日志范围**
我的判断是：必须记录 提交人 / 提交时间 / 筛选条件 / task_id / 下载时间 / 下载 IP / 文件 hash；保留 180 天。请 approve / reject 该清单与天数。

**Q3.4 [确认] 任务幂等窗口**
我的判断是：相同用户 + 相同筛选 + 5 分钟内复用上次结果（E1）。请 approve / reject 5 分钟这个值。

**Q3.5 [确认] 监控告警阈值**
我的判断是：单任务失败立即告警；任务平均耗时 P95 > 10 分钟告警；排队深度 > 10 告警。请 approve / reject。

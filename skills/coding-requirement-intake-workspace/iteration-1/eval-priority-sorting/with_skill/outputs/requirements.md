# Requirements: 订单导出功能

## 1. 背景与业务目标

### 1.1 业务背景

运营团队需要定期从已有 Spring Boot 电商后台系统导出订单数据，用于：

- 财务对账（每月月底高峰）
- 数据分析与运营复盘
- 异常订单排查

当前唯一通道是直接连数据库查询，存在以下问题：

- 效率低：每次都要写 SQL，运营无法自助
- 安全风险大：数据库账号外发、字段越权、查询无审计
- 缺乏并发控制：大查询可能影响线上交易库

### 1.2 业务目标

| 目标编号 | 描述 | 衡量指标 |
|----------|------|----------|
| G1 | 让运营/财务在后台管理界面自助导出订单 Excel | 月底对账期间 0 次找 DBA |
| G2 | 替代直连数据库查询，降低安全风险 | 关闭运营/财务的库账号 |
| G3 | 支持万级数据量稳定导出 | 单次导出 ≤ 50,000 条不超时 |

## 2. In-Scope（MoSCoW 已标注）

- IS-1 [Must]：在后台管理系统新增"订单导出"页面，仅授权角色可见
- IS-2 [Must]：支持按"下单时间范围"筛选订单
- IS-3 [Must]：支持按"订单状态"（多选）筛选订单
- IS-4 [Must]：导出字段固定为 9 列（订单号、下单时间、用户名、商品名、数量、金额、订单状态、收货地址、支付方式）
- IS-5 [Must]：异步导出 + 任务列表 + 完成后下载链接，避免同步阻塞与超时
- IS-6 [Must]：基于现有 RBAC 控制访问权限，仅"运营主管"和"财务"两个角色可用 [ASSUMED：现有系统已存在 RBAC，复用即可]
- IS-7 [Must]：导出操作记录审计日志（操作人、时间、筛选条件、结果文件名、记录数）
- IS-8 [Should]：单次导出最多 50,000 条；超过时拒绝并提示拆分时间段 [ASSUMED：上限基于"几万条"的描述]
- IS-9 [Should]：导出任务列表展示状态（排队中/导出中/成功/失败）和进度百分比
- IS-10 [Should]：失败任务展示错误原因，并提供"重试"按钮
- IS-11 [Should]：下载链接 24 小时有效，过期需要重新生成 [ASSUMED：基于安全合规常规做法]
- IS-12 [Could]：支持按"用户名/订单号"模糊搜索作为附加筛选条件
- IS-13 [Could]：导出文件加水印或在文件名带操作人标识，便于泄漏溯源
- IS-14 [Could]：在导出页面展示历史 30 天导出记录，可重复下载未过期的文件

## 3. Out-of-Scope（[Won't] this time）

- OOS-1 [Won't]：导出 CSV、PDF 或其他非 Excel 格式
- OOS-2 [Won't]：定时/订阅式自动导出（如每日 8:00 自动推送邮件）
- OOS-3 [Won't]：跨系统数据聚合（如订单 + 物流 + 退款合并表）
- OOS-4 [Won't]：开放外部 API 让第三方拉取订单数据
- OOS-5 [Won't]：客户端（移动 App）触发导出
- OOS-6 [Won't]：自定义字段勾选、自定义列顺序
- OOS-7 [Won't]：超过 50,000 条的单次导出（需要分批，由用户拆分时间段）
- OOS-8 [Won't]：实时大屏或 BI 看板替代

## 4. 用户路径与业务流程

### 4.1 主路径（Happy Path）

1. 运营主管登录后台 → 进入"订单管理 → 订单导出"
2. 选择时间范围（必填）+ 订单状态（多选，可全选）
3. 点击"创建导出任务" → 系统校验筛选条件并预估记录数
4. 预估记录数 ≤ 50,000 → 任务提交到异步队列，页面跳到"导出任务列表"
5. 后台 Worker 拉取任务 → 分页查询订单 → 流式写入 Excel → 上传到对象存储 → 更新任务状态为"成功"
6. 用户在任务列表看到状态为"成功"，点击"下载" → 浏览器下载 Excel
7. 系统记录审计日志

### 4.2 关键分支路径

- B1 [Must]：预估记录数 > 50,000 → 直接拒绝并提示"请将时间范围拆为更小区间，单次最多导出 5 万条"
- B2 [Must]：用户角色不在白名单 → 入口菜单不展示，直接访问 URL 返回 403
- B3 [Must]：异步导出任务执行失败（DB 查询超时、磁盘满等）→ 状态置为"失败"，记录错误原因
- B4 [Should]：用户在任务进行中关闭浏览器 → 任务继续后台执行，下次登录可在任务列表看到结果
- B5 [Should]：下载链接过期 → 提示"链接已过期，请重新生成" → 提供"再次导出（相同筛选条件）"按钮
- B6 [Could]：同一用户同时只允许 1 个进行中的导出任务，避免滥用 [ASSUMED]

## 6. 验收标准（Acceptance Criteria）

### 6.1 D-ORDER-EXPORT

- AC-1 [Must]：当授权用户提交有效筛选条件（时间范围必填，订单状态可多选）时，系统应在 2 秒内创建一条 status=PENDING 的 ExportTask 并返回任务 ID。验证方式：接口返回 200 + 任务 ID + 数据库记录。
- AC-2 [Must]：当 ExportTask 被 Worker 拉取后，系统应分页（每页 1000 条）流式查询订单并写入 .xlsx，过程中不应将全量数据加载到内存。验证方式：导出 50,000 条订单时 JVM heap 增量 ≤ 200MB。
- AC-3 [Must]：当导出完成时，系统应将文件上传到对象存储，并将 ExportTask.status 置为 SUCCESS、记录文件 URL 与记录数。验证方式：任务状态机日志 + 文件可下载。
- AC-4 [Must]：当用户提交的筛选条件预估记录数 > 50,000 时，系统应在创建任务前拒绝并返回错误码 EXPORT_OVER_LIMIT。验证方式：单元测试 + 接口返回 4xx + 提示文案。
- AC-5 [Must]：当 ExportTask 执行过程中失败（DB 查询超时、对象存储不可用等），系统应将 status 置为 FAILED 并写入 error_message。验证方式：注入失败的集成测试。
- AC-6 [Should]：当用户在任务列表查看进行中任务时，系统应展示进度百分比（已完成记录数 / 预估记录数 * 100%），刷新间隔不超过 5 秒。验证方式：UI 行为 + 进度字段更新日志。
- AC-7 [Should]：当下载链接生成超过 24 小时后被点击时，系统应返回 410 Gone 并提示"链接已过期，请重新生成"。验证方式：时间穿越测试 + 接口断言。

### 6.2 D-EXPORT-AUDIT

- AC-8 [Must]：每次创建 ExportTask（无论最终成功或失败）都必须写入 ExportAuditLog，包含 user_id、user_name、role、筛选条件 JSON、创建时间。验证方式：DB 行计数 + 字段断言。
- AC-9 [Must]：ExportAuditLog 不可通过任何业务接口删除或修改；仅安全管理员可只读查询。验证方式：代码审查 + 越权测试。

### 6.3 D-EXPORT-AUTH

- AC-10 [Must]：当用户角色不属于 ROLE_OPS_LEAD 或 ROLE_FINANCE 时，"订单导出"菜单不展示，且直接访问 /admin/orders/export 返回 403。验证方式：权限矩阵测试。
- AC-11 [Should]：当授权角色被回收（用户从角色组移除）后，已生成但未下载的链接应失效。验证方式：权限变更后下载链接返回 403 的集成测试。

## 9. 约束与假设

### 9.1 已确认约束（来自 discovery）

- C-1：宿主系统是 Spring Boot，已有订单/用户/商品模块，按其代码风格和模块边界实现。
- C-2：导出字段固定为 9 列，本次不做自定义。
- C-3：使用方仅运营主管 + 财务，约 10 人。
- C-4：常态每天使用，月底对账峰值更高。
- C-5：导出框架由开发选择，未指定品牌。

### 9.2 假设（[ASSUMED]，需用户确认；本测试场景默认 CONFIRMED）

- A-1 [ASSUMED → CONFIRMED]：50,000 条/次 是合理上限（基于"几万条"描述）。
- A-2 [ASSUMED → CONFIRMED]：现有系统已有 RBAC 模型，可复用 ROLE_OPS_LEAD、ROLE_FINANCE。
- A-3 [ASSUMED → CONFIRMED]：下载链接 24 小时失效是常规安全策略。
- A-4 [ASSUMED → CONFIRMED]：导出框架选择 EasyExcel；如团队有别的偏好需在设计阶段调整。
- A-5 [ASSUMED → CONFIRMED]：多商品订单的"商品名"列以分号拼接展示。
- A-6 [ASSUMED → CONFIRMED]：导出走从库读取，避免影响主库交易。
- A-7 [ASSUMED → CONFIRMED]：单用户同时只允许 1 个进行中的导出任务。

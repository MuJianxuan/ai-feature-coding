# Code Review 审查维度

本文档定义 `coding-code-review` 的 6 个审查维度及其检查要点。

## 1. Correctness（正确性）

代码是否正确实现了 design.md 中的方案？

检查要点：
- 业务逻辑是否与 acceptance criteria 一致
- 数据流是否与 design.md 的目标链路匹配
- 边界条件是否正确处理（空值、零值、最大值）
- 错误路径是否返回正确的状态码和错误信息
- 并发场景是否有竞态条件
- 状态机转换是否完整且无死锁

## 2. Security（安全性）

是否存在安全漏洞？

检查要点：
- SQL 注入：是否使用参数化查询
- XSS：用户输入是否正确转义
- 认证/授权：是否有越权访问风险
- 敏感数据：密钥、token 是否硬编码或泄露到日志
- 输入验证：外部输入是否在系统边界验证
- 依赖安全：是否引入已知漏洞的依赖版本
- CSRF：状态变更接口是否有 CSRF 防护

## 3. Performance（性能）

是否存在性能问题？

检查要点：
- N+1 查询：循环内是否有数据库/API 调用
- 内存泄漏：是否有未释放的资源、未取消的订阅
- 不必要的计算：是否有可缓存但重复计算的逻辑
- 大数据量：是否有未分页的列表查询
- 索引：新增查询是否有对应索引
- 阻塞操作：是否在主线程/事件循环中执行耗时操作

## 4. Maintainability（可维护性）

代码是否易于理解和修改？

检查要点：
- 命名：变量、函数、类名是否清晰表达意图
- 函数长度：是否有超过 50 行的函数需要拆分
- 职责单一：每个模块/类是否只做一件事
- 重复代码：是否有可提取的公共逻辑
- 耦合度：模块间依赖是否合理
- 魔法数字：是否有未命名的常量

## 5. Test Coverage（测试覆盖）

关键路径是否有测试？

检查要点：
- 核心业务逻辑是否有单元测试
- 边界条件是否有对应测试用例
- 错误路径是否有测试覆盖
- 集成点（API、数据库）是否有集成测试
- 测试是否真正验证行为（而非只验证调用）
- 测试数据是否有代表性

## 6. Consistency（一致性）

是否与项目现有模式一致？

检查要点：
- 代码风格是否与项目其他部分一致
- 是否遵循项目的 `.docs/spec/coding/` 规范
- 错误处理模式是否与项目约定一致
- API 响应格式是否与现有接口一致
- 目录结构是否遵循项目约定
- 依赖使用是否与项目已有选择一致（不引入重复功能的新依赖）

## 严重程度判定标准

| 级别 | 标准 | 示例 |
| --- | --- | --- |
| BLOCKING | 会导致数据丢失、安全漏洞、核心功能错误 | SQL 注入、越权访问、数据覆盖 |
| WARNING | 会导致性能问题、可维护性下降、潜在 bug | N+1 查询、缺少测试、过度耦合 |
| NITPICK | 风格偏好、微小优化、非关键改进 | 命名可更清晰、可用更简洁写法 |

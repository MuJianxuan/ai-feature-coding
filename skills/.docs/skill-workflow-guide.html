新建功能说清楚目标和资料状态即可。没有资料时，orchestrator 必须先让用户在 PRD 直通、产品提供、零到一、迭代增强之间选择，不能自行猜测。

入口场景

NEW_FEATURE 启动前先判定 scenario。无资料入口不能猜测模式；如果已有未完成 feature，也必须询问继续还是新建。

A 零到一只有一句话想法，无 PRD、原型或代码基线。进入 ship-discover greenfield；涉及 UI 且无设计稿时再进 ship-shape。
Discover 激活
B 产品提供已有 PRD、Figma、原型或 UIUX 材料，但允许继续澄清缺口。跳过 Discover，ship-define 走 interview mode。
Define 起步
C 迭代增强基于已有 feature 目录、代码路径或现有功能做变更。必须有现状基线，否则先问清楚增强对象。
Discover evolve
D PRD 直通完整 PRD + 原型/设计稿，且用户明确不需要需求录入。跳过 Discover，ship-define 走 prd_direct。

Scope 与模式

scenario 决定是否进入 Discover；workspace_mode 决定单项目还是多项目组；project_scope 决定前后端阶段和测试轨道裁剪。standard 是默认模式，fast-track 只能在规则允许时压缩路径。

fullstack默认范围。保留 contract、frontend design、backend design、双 plan、全量测试轨道。
backend_only跳过 ship-frontend-design；A/C 场景下 ship-shape 也自动跳过。测试只保留后端轨。
frontend_only跳过 ship-backend-design；测试只保留前端 component / E2E 轨。
project_groupfeature 目录写入 workspace feature_root；projects: [web, api] 记录默认关联项目，允许经用户确认临时扩展。
standard默认完整路径。所有硬门禁必须通过，设计、计划、验证产物完整留痕。
fast-track小型功能或紧急修复可用。不能绕过需求定义、需求评审、测试和验收；复杂 UI 要升级回 standard。
Contract-First实施顺序固定：契约层任务 → 前后端实现 → 集成联调。正式 build 始终单 DOING。

产物与门禁

普通产物看 stage_status；评审产物看 review_status、user_sign_off 和 signed_at。

Feature 目录<workspace>/.docs/feature-YYYYMMDD-<name>/。包含 meta.yml、阶段文档和 resource/ 原始资料。
普通产物 frontmatterstage、artifact_role、stage_status、updated_at、evidence_complete 是最小契约。
meta.yml只做恢复、汇总和路由索引。冲突时信任阶段文档 frontmatter，再回写修正。
skip_log记录用户显式强制跳过软门禁或默认前置阶段的审计事实，不改变 stage status 语义。
Requirementsrequirements.md 是 Define 输出；场景 D 是 PRD 索引式提取，不复制原文，不做需求采访。
Design Packapi-contract.md 是共享基线；前端、后端设计可以并行，但必须回到 review-design.md 收口。
Delivery Planship-delivery-plan 阶段内固定 frontend -> backend -> sync；正式 build 仍保持单 DOING。
Traceabilityvalidate_traceability.py 检查 AC、Domain ID、contract、plan 和 test evidence 的可追溯关系。
Evidence Packvalidate_verification.py 与 validate_handoff.py 约束测试轨道结果、验收证据和残余风险签收。
Define Reviewreview-define.md 确认需求清晰、AC 可验证、无关键歧义。
Design Reviewreview-design.md 交叉检查 contract ↔ frontend ↔ backend 的一致性。
Plan Reviewreview-plan.md 审查任务粒度、依赖顺序和 AC 全覆盖。
Shared Gate Referencereview-gate-reference.md 统一三道 gate 的 frontmatter、decision、finding、required changes 与 sign-off 语义。
Machine-Readable Contractship-contract 要求 REST / gRPC / message / CLI / SDK 等契约形态提供可机器校验策略。
Task Readinessbuild_task_preflight.py 在进入正式 build 前检查单个 DOING 任务是否有 AC、contract、依赖和验证路径。
Regression Promptsship-orchestrator/tests/regression-prompts.md 覆盖产品提供入口、未签 gate、需求质量、DAG、preflight 和 handoff 场景。

Delegation Contract

子代理是执行策略，不是新阶段。只有主上下文可以推进 current_stage、修改 meta.yml 和写入用户签字。

forbiddenship-contract、ship-tech-discovery.selection、ship-delivery-plan 不委派，保持单一决策链。
parallel_owned_outputs仅前后端设计。子代理可拥有 frontend-design.md 或 backend-design.md，但不能推进 gate。
gate_check_switchable三道 hard gate 可由子代理起草检查文档，但必须保持 review_status: pending。
assistive_onlyresearch 取证、资料索引、测试分轨、证据整理。不能改正式产物 frontmatter。
user_gate_only批准、拒绝、继续下一个 build slice、关闭 feature、是否写回 spec proposal 都由用户决策。
合法执行值node_overrides 只接受 current_context、assistive_subagent、parallel_subagent、gate_check_subagent。
Canonical Nodebuild 辅助节点包含 ship-build.read-next-task、ship-build.spec-scan、ship-build.env-precheck、ship-build.evidence-pack。
Verify 轨道ship-verify.backend-unit、integration、contract、frontend-component、frontend-e2e 可辅助分轨，主上下文更新 verification.md。
Warning Log委派解析告警写入 meta.yml.delegation.warnings，不得复用 spec_context.warnings。

Workspace Spec

ship-spec 是 workflow utility，不是 stage。orchestrator 负责解析 workspace；阶段 skill 负责读取规范并在产物中留痕。

边界显式配置源是 <workspace>/.docs/ship/project.yml。默认 spec_root 是 workspace 下的 .docs/spec。
Warn Then Continue缺 INDEX.md、无匹配规范或 frontmatter 不合法时写 warning，默认继续；无法确定 workspace 才阻塞。
Hook 点ship-tech-discovery、前后端设计、ship-build、ship-handoff 会消费或汇总规范；ship-spec 不写入 stage map。
运行 helperhelper 负责 init-workspace / scan / resolve / sync，不替阶段 skill 写正式产物。多项目父目录下必须先初始化 workspace config，再为 feature 选择 projects。
python3 skills/ship-orchestrator/scripts/feature_meta_runtime.py init-workspace <workspace> --workspace-mode project_group --workspace-name demo --project web --project api
python3 skills/ship-orchestrator/scripts/spec_runtime.py scan --project-config <workspace>/.docs/ship/project.yml
python3 skills/ship-orchestrator/scripts/feature_meta_runtime.py sync-spec <feature>/meta.yml --project-config <workspace>/.docs/ship/project.yml --stage ship-build --file web/src/app.ts
Proposal-Firstship-handoff 只记录待沉淀 proposal；真正写回 workspace spec_root 需要用户确认。

恢复与状态查询

恢复时优先报告 macro stage；只有诊断、阻塞或高级调用时才展开内部阶段。

CONTINUE读取 feature meta.yml，优先信任已有 spec_context.workspace_*，再根据 stage status 路由。
INSPECT扫描 workspace feature_root/feature-*，过滤 active / blocked，输出功能名、当前大阶段、更新时间和阻塞原因。
冲突处理meta.yml 与产物 frontmatter 不一致时，优先信任产物；孤立产物或阶段缺失要报告并询问处理。
资料准备态场景 B/D 先创建 raw inbox 时，ship-define 可为 blocked + awaiting_materials；资料放入后解除阻塞。
Project Reality First已有项目进入 Design 前，ship-tech-discovery 必须先查真实代码、表、API、页面、服务、权限和旧 feature 文档。
Scope 冻结project_scope 只允许在 ship-design-review 前变更；设计评审通过后冻结。

可执行护栏

主要自然语言协议已补成可运行检查。修改 workflow、SKILL.md、模板、阶段产物或本指南时，优先用这些脚本验证。

Artifact Lintvalidate_feature_artifacts.py 做统一 frontmatter、stage status 和阶段产物入口检查。
Stage Transitionstage_transition_check.py 判断目标阶段是否允许推进，避免绕过 hard gate 或未 ready 产物。
Requirementsvalidate_requirements.py 检查 Domain ID、AC、NFR、open questions、模糊词和 ready 条件。
Discovery / UIvalidate_product_brief.py、validate_ui_artifacts.py 检查 confidence、evidence、variants、wireframe 和 visual token 信号。
Tech / Contractvalidate_tech_discovery.py、validate_contract.py 检查 source_id、ADR、rejected alternatives、schema、error 和 AC mapping。
Design Alignmentvalidate_frontend_design.py、validate_backend_design.py、validate_design_alignment.py 检查 page/API/state、endpoint implementation、migration/rollback 和 contract 对齐。
Plan / Build / Closevalidate_delivery_plan.py、build_task_preflight.py、validate_verification.py、validate_handoff.py 检查 DAG、单 DOING、测试轨道和验收证据。

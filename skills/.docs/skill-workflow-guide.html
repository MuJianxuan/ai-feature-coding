ShipKit Skills Workflow · 2026-05-29 Snapshot

一个入口，五个大阶段，十四个内部节点，再加一条 project-local 规范边界。

当前这套 skills workflow 的最新高信心口径是：使用者默认只面对 ship-orchestrator 和 [Discover →] Define → Design → Build → Close； 维护者则通过 canonical stage、frontmatter、hard gate、delegation contract， 以及 target project 驱动的 ship-spec 解析协议，保持流程可恢复、可审计、可验证。

启动 ship-orchestrator，我想做一个“功能名”：一句话想法
python3 skills/ship-orchestrator/scripts/spec_runtime.py scan --project-config <target-project>/.docs/ship/project.yml
默认入口 用户默认只和 ship-orchestrator 交互，由它识别 NEW / CONTINUE / INSPECT。
规范边界 ship-spec 只消费 target project 的 spec_root，不再默认为 cwd /.docs/spec。
三道硬门禁 Define Review、Design Review、Plan Review 必须 approved + 用户签字。
委派原则 子代理是执行策略，不是新阶段；正式状态推进和用户签字仍由主上下文收口。

使用者视图：四种入口场景

NEW_FEATURE 先识别场景，再决定是否进入 Discover。场景 B/D 跳过 Discover；A/C 激活 Discover。CONTINUE 与 INSPECT 则围绕 feature 目录恢复与汇总。

A 零到一

只有一句话想法，无 PRD / 原型 / 设计稿。先做需求探索，必要时再进入 ship-shape 补 UIUX 原型。

start: ship-discover

B 产品提供

已有 PRD / Figma / prototype / UIUX 材料，但仍需要整理和采访确认。若资料未放齐，会进入 awaiting_materials。

start: ship-define

C 迭代增强

基于已有 feature 或代码路径做变更。先扫描现状、分析影响，再生成 product brief。

start: ship-discover

D PRD 直通

已有完整 PRD + 原型 / 设计稿，并明确不需要需求录入。ship-define 走 prd_direct，零提问纯提取。

start: ship-define

状态事实源与恢复索引

阶段产物决定状态，orchestrator 只缓存摘要并负责路由。最新口径下，target project snapshot 也进入了 meta.yml.spec_context。

阶段文档 frontmatter 非评审产物使用 stage_status: draft | ready | complete。达到 ready 才能通过 soft gate。
meta.yml 保存 current_stage、macro_stage、scenario、delegation、spec_context 等恢复索引。
spec_context snapshot 新增 target_project_id、target_project_root、spec_root、feature_root、resolution_source。
冲突处理 如果 meta.yml 与产物 frontmatter 不一致，优先信任产物文档，再回写修正 meta.yml。

Project-Local Spec Boundary

这是当前 guide 最大的新增重点。ship-spec 不再围绕 cwd 推断，而是显式绑定 target project。

唯一显式配置源 target project/.docs/ship/project.yml 是 project boundary 的唯一显式配置源。默认声明 project_root、spec_root、feature_root，本期只支持 project_level_only。
两种可信来源 第一来源是当前 feature 已持久化的 meta.yml.spec_context；第二来源是显式 .docs/ship/project.yml。不再根据 package.json、go.mod 或父目录结构 silent guess。
阻塞与 warning 只有无法确定 target project 时才阻塞。target project 已确定但缺少 spec_root、INDEX.md、命中 spec 或 frontmatter 合法性时，统一走 Warn Then Continue。

关键 helper 与恢复路径

当前高信心口径下，helper 也分成了显式 project-config 模式与已落盘 meta 模式。

spec_runtime.py scan / resolve 默认优先走 --project-config。只有低层调试时，才显式传 --spec-root + --project-root。 
python3 skills/ship-orchestrator/scripts/spec_runtime.py
scan --project-config <target-project>/.docs/ship/project.yml
feature_meta_runtime.py init NEW_FEATURE 会先解析 target project，再把 feature 创建到该项目自己的 feature_root。如果 feature 路径本身就在 target project 下，也允许自动上溯发现 project config。 
python3 skills/ship-orchestrator/scripts/feature_meta_runtime.py
init feature-YYYYMMDD-demo
--project-config <target-project>/.docs/ship/project.yml
--feature-name Demo --feature-id feature-YYYYMMDD-demo
--scenario product_provided
feature_meta_runtime.py sync-spec CONTINUE_FEATURE 优先信任已落盘的 meta.yml.spec_context.target_project_*。若 meta 没有 target project 信息，则必须显式提供 --project-config。 
python3 skills/ship-orchestrator/scripts/feature_meta_runtime.py
sync-spec <target-project>/.docs/feature-YYYYMMDD-demo/meta.yml
--stage ship-build --file src/app.ts

三道 hard gate

hard gate 不可跳过。只有 review_status: approved 且 user_sign_off / signed_at 非空，才能继续。

ship-define-review 检查需求目标、用户路径、业务域、AC、非功能需求、约束和资料一致性。
ship-design-review 交叉验证 api-contract.md、frontend-design.md、backend-design.md 的接口、字段和数据流。
ship-plan-review 检查任务粒度、依赖关系、关键路径、前后端并行可行性和 AC 覆盖。

子代理委派边界

委派是执行策略，不改变 stage。只有主上下文能修改 meta.yml、写最终签字、推进 current_stage；并且 hard gate / parallel / assistive 的解释规则都已固化到 protocol。

forbidden 共享契约、技术选型收口、交付计划同步等节点不可委派，避免决策链分叉。
assistive_only 可做资料搜集、审计、测试分轨、证据整理；不能直接改正式 frontmatter 或 meta.yml。
parallel_owned_outputs 仅前端设计和后端设计可并行拥有正式产物，并共享同一份 API contract 与 spec_context。
gate_check_switchable 子代理可起草 review 文档，但必须保持 pending，最终结论由主上下文复核并向用户请求签字。
user_gate_only 用户确认、继续/关闭、accept / follow-up / proposal 取舍不能委派。

ShipKit Solo

个人开发者通用开发流程技能套件

新版 ShipKit 保留 ship-* 命名和 ship-orchestrator 入口，但默认流程已重构为轻量、证据驱动、一次一个 slice 的 solo developer workflow。

旧版多重审批关和签字 preflight 不再是默认流程；review skills 作为可选 checklist 使用。

默认入口

ship-orchestrator 负责识别任务模式、压缩阶段、路由支持技能、恢复上下文和交付总结。

  • feature：从 Discover/Define 开始。
  • bugfix：通常从 Understand 开始。
  • refactor：用 invariant contract 防止行为漂移。
  • ui：必要时插入 ship-shape。
  • docs/release：自动跳过不必要设计阶段。

轻量门禁

默认门禁是证据检查，不是审批系统。

  • Scope：目标、非目标、AC 是否清楚。
  • Reality：是否读过相关代码、测试、配置和文档。
  • Slice：下一步是否足够小，文件范围是否明确。
  • Evidence：验证命令和结果是否能映射到 AC。
  • Close：风险、follow-up 和停止条件是否说明。

工作目录

默认产物写入 .docs/ship/<work-id>/。

  • meta.yml：轻量索引。
  • brief.md：目标、范围、AC。
  • context-map.md：仓库事实。
  • contract.md：最小边界或不变量。
  • plan.md：单 slice 执行计划。
  • verification.md / handoff.md：验证和交付。

技能职责

Skill职责
ship-discover澄清真实目标、约束、非目标和成功信号。
ship-define生成轻量 brief、AC、假设和风险。
ship-tech-discovery读取仓库事实，避免凭猜测设计或编码。
ship-contract定义 API、事件、CLI、UI state、数据或行为不变量。
ship-delivery-plan拆成可验证 slices，列 allowed files 和 verification command。
ship-build只实现当前 DOING slice。
ship-verify映射 AC 到测试、构建、截图或手工验证证据。
ship-handoff总结改动、验证、风险和下一步。
ship-*-review可选 checklist，不写入 runtime stage，不默认阻塞。

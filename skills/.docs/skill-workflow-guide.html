ShipKit Skills Workflow

一个入口，五个大阶段，十四个内部执行节点。

当前项目是一套从需求到交付的 development workflow skills。默认使用者只需要面对 ship-orchestrator 和 [Discover →] Define → Design → Build → Close， 维护者则通过 canonical stage、frontmatter、hard gate 与 delegation contract 保持流程可恢复、可审计、可验证。

启动 ship-orchestrator，我想做一个“功能名”：一句话想法
默认入口 用户默认只和 ship-orchestrator 交互，由它识别 NEW / CONTINUE / INSPECT。
事实源 阶段文档 frontmatter 是事实源，meta.yml 是恢复与展示索引。
三道硬门禁 Define Review、Design Review、Plan Review 必须 approved + 用户签字。
委派原则 子代理是执行策略，不是新阶段；状态推进仍由主上下文收口。

使用者视图：四种入口场景

NEW_FEATURE 先识别场景，再决定是否进入 Discover。场景 B/D 跳过 Discover，场景 A/C 激活 Discover。

A 零到一

只有一句话想法，无 PRD、原型、设计稿。先进入需求探索，必要时再进入 UIUX 原型设计。

start: ship-discover

B 产品提供

已有 PRD / Figma / prototype / UIUX 材料，但仍需要整理和采访确认。

start: ship-define

C 迭代增强

基于已有 feature 或代码路径做变更。先扫描现状、分析影响，再生成 product brief。

start: ship-discover

D PRD 直通

已有完整 PRD + 原型/设计稿，并明确不需要需求录入。ship-define 走 prd_direct。

start: ship-define

状态事实源与推进规则

这是恢复断点和避免多套状态冲突的核心。阶段产物决定状态，orchestrator 只缓存摘要并负责路由。

阶段文档 frontmatter 非评审产物使用 stage_status: draft | ready | complete。达到 ready 才能通过 soft gate。
meta.yml 保存 current_stage、macro_stage、scenario、delegation、spec_context 等恢复索引。
冲突处理 如果 meta.yml 与产物 frontmatter 不一致，优先信任产物文档，再回写修正 meta.yml。

三道 hard gate

hard gate 不可跳过。只有 review_status approved 且 user_sign_off / signed_at 非空，才能继续。

ship-define-review 检查需求目标、用户路径、业务域、AC、非功能需求、约束和资料一致性。
ship-design-review 交叉验证 api-contract.md、frontend-design.md、backend-design.md 的接口、字段和数据流。
ship-plan-review 检查任务粒度、依赖关系、关键路径、前后端并行可行性和 AC 覆盖。

子代理委派边界

委派是执行策略，不改变 stage。只有主上下文能修改 meta.yml、写最终签字、推进 current_stage。

forbidden 共享契约、技术选型收口、交付计划同步等节点不可委派，避免决策链分叉。
assistive_only 可做资料搜集、审计、测试分轨、证据整理；不能直接改正式 frontmatter。
parallel_owned_outputs 仅前端设计和后端设计可并行拥有正式产物。
gate_check_switchable 子代理可起草 review 文档，但必须保持 pending，最终结论由主上下文确认。
user_gate_only 用户确认、继续/关闭、accept/follow-up/proposal 取舍不能委派。

ShipKit Skills Workflow · 2026-05-30 Snapshot

一个入口，五个大阶段，十四个内部阶段。

默认从 ship-orchestrator 进入。用户看到的是 [Discover →] Define → Design → Build → Close；内部用 canonical stage、frontmatter、三道 hard gate、project-local ship-spec 和 delegation contract 保证流程可恢复、可审计、可验证。

Scope 与模式

scenario 决定是否进入 Discover；project_scope 决定前后端阶段和测试轨道裁剪。standard 是默认模式，fast-track 只能在规则允许时压缩路径。

fullstack默认范围。保留 contract、frontend design、backend design、双 plan、全量测试轨道。
backend_only跳过 ship-frontend-design；A/C 场景下 ship-shape 也自动跳过。测试只保留后端轨。
frontend_only跳过 ship-backend-design；测试只保留前端 component / E2E 轨。
standard默认完整路径。所有硬门禁必须通过，设计、计划、验证产物完整留痕。
fast-track小型功能或紧急修复可用。不能绕过需求定义、需求评审、测试和验收；复杂 UI 要升级回 standard。
Contract-First实施顺序固定：契约层任务 → 前后端实现 → 集成联调。正式 build 始终单 DOING。

产物与门禁

普通产物看 stage_status；评审产物看 review_status、user_sign_off 和 signed_at。

Feature 目录<target-project>/.docs/feature-YYYYMMDD-<name>/。包含 meta.yml、阶段文档和 resource/ 原始资料。
普通产物 frontmatterstage、artifact_role、stage_status、updated_at、evidence_complete 是最小契约。
meta.yml只做恢复、汇总和路由索引。冲突时信任阶段文档 frontmatter，再回写修正。
skip_log记录用户显式强制跳过软门禁或默认前置阶段的审计事实，不改变 stage status 语义。
Define Reviewreview-define.md 确认需求清晰、AC 可验证、无关键歧义。
Design Reviewreview-design.md 交叉检查 contract ↔ frontend ↔ backend 的一致性。
Plan Reviewreview-plan.md 审查任务粒度、依赖顺序和 AC 全覆盖。

Delegation Contract

子代理是执行策略，不是新阶段。只有主上下文可以推进 current_stage、修改 meta.yml 和写入用户签字。

forbiddenship-contract、ship-tech-discovery.selection、ship-delivery-plan 不委派，保持单一决策链。
parallel_owned_outputs仅前后端设计。子代理可拥有 frontend-design.md 或 backend-design.md，但不能推进 gate。
gate_check_switchable三道 hard gate 可由子代理起草检查文档，但必须保持 review_status: pending。
assistive_onlyresearch 取证、资料索引、测试分轨、证据整理。不能改正式产物 frontmatter。
user_gate_only批准、拒绝、继续下一个 build slice、关闭 feature、是否写回 spec proposal 都由用户决策。

Project-Local Spec

ship-spec 是 workflow utility，不是 stage。orchestrator 负责解析 target project；阶段 skill 负责读取规范并在产物中留痕。

边界显式配置源是 <target-project>/.docs/ship/project.yml。默认 spec_root 是 target project 下的 .docs/spec。
Warn Then Continue缺 INDEX.md、无匹配规范或 frontmatter 不合法时写 warning，默认继续；无法确定 target project 才阻塞。
Hook 点ship-tech-discovery、前后端设计、ship-build、ship-handoff 会消费或汇总规范。
运行 helperhelper 负责 scan / resolve / sync，不替阶段 skill 写正式产物。多项目父目录下必须先显式选定 project config。
python3 skills/ship-orchestrator/scripts/spec_runtime.py scan --project-config <target-project>/.docs/ship/project.yml
python3 skills/ship-orchestrator/scripts/feature_meta_runtime.py sync-spec <feature>/meta.yml --project-config <target-project>/.docs/ship/project.yml --stage ship-build --file src/app.ts
Proposal-Firstship-handoff 只记录待沉淀 proposal；真正写回 target project spec_root 需要用户确认。

恢复与状态查询

恢复时优先报告 macro stage；只有诊断、阻塞或高级调用时才展开内部阶段。

CONTINUE读取 feature meta.yml，优先信任已有 spec_context.target_project_*，再根据 stage status 路由。
INSPECT扫描 target project feature_root/feature-*，过滤 active / blocked，输出功能名、当前大阶段、更新时间和阻塞原因。
冲突处理meta.yml 与产物 frontmatter 不一致时，优先信任产物；孤立产物或阶段缺失要报告并询问处理。
